# 사용법 (USAGE)

이 키트가 제공하는 **슬래시 커맨드**·**서브에이전트**의 용도·호출법·예시 모음입니다.
한 줄 요약은 `CLAUDE.md`의 "쓸 수 있는 도구"에, 복사 방법은 `README.md`에 있습니다.

> **커맨드 vs 에이전트**
> - **슬래시 커맨드(`/이름`)** = 메인 Claude에게 주는 정해진 지시 묶음. 직접 입력해 호출.
> - **서브에이전트** = 특정 역할의 독립 작업자. 메인 Claude가 `Task`로 위임하거나, 사용자가 "○○ 에이전트로 …" 라고 요청하면 호출.

---

## 슬래시 커맨드 (5)

### `/plan [작업 설명]`
**용도**: 코드 수정 **전에** 구현 계획을 세운다. 관련 파일 탐색 → 변경 파일·순서 단계화 → 위험/되돌리기 어려운 지점 표시 → 승인 후 실행.
**언제**: 멀티파일 변경, 영향 범위가 넓거나 불확실한 작업을 시작하기 전.
**예시**: `/plan 리프레시 토큰 회전에 재사용 탐지 추가`
**참고**: 더 깊은 대화형 계획이 필요하면 `impl-planner`/`product-planner` 에이전트로.

### `/review`
**용도**: 현재 작업 트리의 변경분(또는 최근 수정 코드)을 **심각도별**로 리뷰. critical(버그/보안) / major(설계) / minor(스타일). 직접 고치지 않고 지적만.
**언제**: 코드를 짠 뒤(작성·리뷰 분리 원칙). 커밋 전 점검.
**예시**: `/review`
**참고**: 규모가 크면 `code-reviewer` 에이전트에 위임.

### `/verify`
**용도**: 방금 변경이 **실제로 동작하는지** 검증. 테스트/린트/타입체크 실행 결과를 그대로 보여주고, 앱이면 돌려서 확인. "통과" 선언 전 증거 제시.
**언제**: 변경 직후, "됐다"고 말하기 전.
**예시**: `/verify`
**`/ultraqa`와 차이**: `/verify`는 **1회** 검증·보고. `/ultraqa`는 통과까지 **반복**(진단·수정 포함).

### `/okf [범위]`
**용도**: 프로젝트 지식을 **OKF 번들**(`knowledge/`)로 자동 생성·갱신. `knowledge-curator`에게 위임해 코드 근거로 개요·DB·API 등 개념 문서를 만든다. 추측 금지.
**언제**: DB 스키마·API·아키텍처에 **의미 있는 변경**이 생겼을 때.
**예시**: `/okf` (전체) · `/okf api` (API 개념만) · `/okf database`
**산출**: `knowledge/<폴더>/<개념>.md` + `index.md`(목차) + `log.md`(이력) 갱신.

### `/ultraqa [--옵션]`
**용도**: 목표가 통과할 때까지 **검증 → 진단 → 수정 자동 반복**(최대 5회, 동일 실패 3회면 조기중단).
**옵션**: `--tests` · `--build` · `--lint` · `--typecheck` · `--custom "패턴"` · `--interactive`(앱 띄워 `qa-tester`로 검증). 비우면 대화로 목표 확인.
**언제**: "고칠 게 뭔지는 아는데 통과만 시키면 됨" — 게이트를 자동으로 녹색 만들 때.
**예시**: `/ultraqa --typecheck` · `/ultraqa --build` · `/ultraqa --interactive`
**구성**: 진단은 `debugger`/`architect`, 인터랙티브 검증은 `qa-tester`를 내부에서 사용.

## 서브에이전트 (8)

호출: 메인 Claude가 자동 위임하거나, **"○○ 에이전트로 …"** 라고 직접 요청.

### 계획 단계 (읽기 전용 — 코드 안 건드림)
| 에이전트 | 역할 | 언제 위임 |
|---|---|---|
| `analyst` | 요구사항 정리. **질문으로만** 정보 수집(목적·이해관계자·모호함·성공기준). | 무엇을 만들지 자체가 불명확할 때 |
| `product-planner` | 제품/기능 기획. 범위·우선순위·MVP·마일스톤을 사용자 답변으로 확정. | 기능 스코프·우선순위를 정할 때 |
| `impl-planner` | 구현 계획. 단계·의존·위험·검증 정리, 불명확한 기술 결정은 질문. | 어떻게 만들지를 단계화할 때 |
| `architect` | 설계·구조·디버깅 **전략** 자문. 트레이드오프 검토. | 큰 변경 전 방향 잡기 |

### 구현·품질 단계
| 에이전트 | 역할 | 언제 위임 |
|---|---|---|
| `code-reviewer` | 변경 코드 심각도별 리뷰(버그·SOLID·성능). 고치지 않고 지적만. | 멀티파일 변경 후 / `/review` 위임 대상 |
| `debugger` | 버그·에러·테스트 실패의 **근본 원인** 추적(스택트레이스·회귀·빌드 에러). | 원인을 모를 때 |
| `qa-tester` | 앱을 **실제로 띄워** 동작 검증(기동·통합·사용자 대면). 구현/수정은 안 함. | "진짜 실행하면 되나?" 확인 / `/ultraqa --interactive` 내부 |
| `knowledge-curator` | 지식을 OKF 번들로 생성·갱신. 코드 근거만. | `/okf` 위임 대상 |

> `qa-tester`는 플랫폼 적응형: mac/linux는 tmux, **Windows는 백그라운드 프로세스 + 포트 폴링**으로 동작.

## 전형 워크플로우

**새 기능 (불명확 → 출시)**
```
analyst (요구 확정)  →  product-planner / impl-planner (계획)
   →  /plan (착수 전 점검)  →  구현
   →  /review (작성·리뷰 분리)  →  /verify 또는 /ultraqa (통과까지)
   →  /okf (지식 문서 갱신)
```

**버그 잡기**
```
debugger (근본원인)  →  수정  →  /ultraqa --tests (회귀까지 막고 통과 확인)
```

**게이트 녹색 만들기 (빌드/타입/린트 깨짐)**
```
/ultraqa --build   (실패 → debugger 진단 → 수정 → 재검증, 통과까지 반복)
```
